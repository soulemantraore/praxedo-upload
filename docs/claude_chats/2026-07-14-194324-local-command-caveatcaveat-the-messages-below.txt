  Design — Monorepo praxedo-upload

  1. Cible

  Un nouveau dépôt git praxedo-upload créé dans un dossier neuf
  (~/Documents/praxedo-upload/), poussé vers
  github.com/soulemantraore/praxedo-upload. On travaille dans un dossier neuf
  pour ne rien casser pendant la migration : les 3 dossiers actuels restent
  intacts jusqu'à validation.

  2. Structure

  praxedo-upload/
  ├── .github/workflows/
  │   ├── backend-deploy.yml     # paths: praxedo-upload-backend/**
  │   ├── ui-deploy.yml          # paths: praxedo-upload-ui/**
  │   └── scanner-deploy.yml     # paths: praxedo-upload-scanner/**
  ├── praxedo-upload-backend/    # subtree, historique préservé
  ├── praxedo-upload-scanner/    # subtree, historique préservé
  ├── praxedo-upload-ui/         # subtree, historique préservé
  ├── docs/                      # documentation projet
  ├── .claude/                   # mémoire + sessions (versionnées)
  ├── CLAUDE.md                  # mis à jour : monorepo
  ├── README.md                  # NOUVEAU : porte d'entrée du livrable
  └── .gitignore                 # racine (.DS_Store, .superpowers/,
  .worktrees/, dumps)

  3. Migration d'historique (via git subtree, tout préservé)

  Phase 0 — consolidation par composant (pour ne rien perdre) :
  - backend : committer les 2 fichiers deploy/*, vérifier/intégrer
  task/remote-scanner-client, consolider sur une branche unique.
  - ui : merger develop ↔ task/frontend-deploy (résoudre la divergence 1/1).
  - scanner : rien à faire, main est propre.

  Puis git subtree add --prefix=praxedo-upload-<x> <chemin-local> 
  <branche-consolidée> pour chacun → chaque sous-dossier garde son historique
  complet (git log -- praxedo-upload-backend/ fonctionnera).

  4. Anciens dépôts

  Les 3 repos GitHub (-backend, -ui, -scanner) sont archivés (lecture seule) une
  fois la migration validée — sauvegarde intégrale de l'historique d'origine.

  5. Déploiement séparé (CI path-filtered) — simple et straightforward

  3 workflows GitHub Actions à la racine, chacun avec un filtre paths:. Un push
  qui ne touche que praxedo-upload-ui/** ne redéclenche que le déploiement UI.
  On repart des 2 deploy.yml existants (backend, ui) qu'on adapte, et on en crée
  un pour le scanner.

  6. CLAUDE.md

  Mise à jour des sections « Structure du dépôt » (polyrepo → monorepo), « Git /
  workflow » et « Pièges connus ».
