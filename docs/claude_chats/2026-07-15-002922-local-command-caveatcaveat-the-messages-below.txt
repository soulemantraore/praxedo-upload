⏺ J'ai tout le contexte. Point important : la mémoire .claude/memory/infra.md
  disait que l'api devait rester en YAML Knative parce qu'elle embarquait un 
  sidecar Cloud SQL Auth Proxy (donc multi-conteneurs). Or ce n'est plus vrai :
  depuis la migration vers Supabase en TLS direct, api et worker sont 
  mono-conteneur (un seul container app dans les deux YAML). Le verrou
  historique a donc sauté — gcloud run deploy mono-conteneur marche parfaitement
  pour les deux.

  Voici ce que je compte faire (backend deploy/Makefile) :

  deploy-api → remplacer render-api + gcloud run services replace par :
  gcloud run deploy $(API_SERVICE) --image=$(IMAGE_URI) --region --project \
    --service-account=$(API_SA) --port=8080 --cpu=1 --memory=512Mi \
    --min-instances=0 --max-instances=10 --timeout=60 --ingress=all \
    --labels=app=praxedo-upload,component=api \
    --allow-unauthenticated \
    --set-env-vars=SPRING_PROFILES_ACTIVE=gcp,SERVER_PORT=8080,GCP_PROJECT_ID=..
  .,GCS_BUCKET=...,PUBSUB_SCAN_TOPIC=...,DB_URL=...,DB_USER=... \
    --set-secrets=DB_PASSWORD=$(DB_SECRET_NAME):latest
  --allow-unauthenticated remplace le binding allUsers séparé (je le retire
  donc).

  deploy-worker → idem, en gardant le garde-fou SCANNER_URL (échoue si le
  scanner n'est pas déployé), avec --no-allow-unauthenticated (reste privé, seul
  le push SA obtient run.invoker via pubsub-push), --memory=1Gi 
  --max-instances=5 --timeout=600, et les env SCANNER_URL/AUDIENCE/OIDC.

  Je supprime aussi les cibles render-api/render-worker, mets à jour .PHONY + le
  help, et le README backend.

  Note : les startupProbe/livenessProbe TCP explicites du YAML disparaissent —
  Cloud Run applique de toute façon une sonde de démarrage TCP par défaut sur le
  port. Comportement quasi identique.

  Comment make importe les variables

  Dans les Makefiles, toutes les variables de config sont en ?= (ex. PROJECT_ID 
  ?= …, DB_URL ?= …). Ordre de priorité pour une variable ?= :

  1. make VAR=… en ligne de commande (gagne toujours) →
  2. variable d'environnement VAR du même nom (le bloc env: du job) →
  3. valeur par défaut du Makefile.

  Donc une variable n'arrive dans la recette que si le CI la passe en argument
  make ou la met dans env:. Sinon → valeur par défaut du Makefile.

  Scanner (scanner-deploy.yml) — ✅ complet

  make build-push deploy reçoit PROJECT_ID, REGION, IMAGE_TAG. La recette deploy
  ne consomme que ça + des dérivées (SCANNER_SA, IMAGE_URI) et CLAMAV_IMAGE
  (défaut sain). Rien ne manque.

  UI_ORIGIN.
