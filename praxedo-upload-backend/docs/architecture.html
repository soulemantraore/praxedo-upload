Documentation d'architecture

Micro-service de fichiers securises praxedo-upload-backend

Une regle gouverne toute l'architecture : aucun fichier n'est servi sans avoir ete valide par un antivirus. Le reste des choix decoule de cet invariant, et de la contrainte d'echelle (nombreux clients simultanes, fichiers de tailles tres variables).

Java 21 Spring Boot 3.3 Architecture hexagonale GCP / Cloud Run Scanner Python + ClamAV PostgreSQL + Flyway Pub/Sub
§1  Vue d'ensemble

Recevoir, mettre en quarantaine, ne servir que le sain

Le service recoit des fichiers (utilisateurs et systemes tiers), garantit qu'ils sont analyses par un antivirus, et n'expose au telechargement que ceux declares sains. Un fichier infecte reste en quarantaine.

Architecture hexagonale : un domaine pur entoure de ports dont les adapters sont choisis par le profil Spring actif. Le meme code metier tourne en local sans dependance externe et en production sur GCP — on ne remplace que les adapters.

CLEAN — valide, telechargeable INFECTED — bloque, quarantaine PENDING / SCANNING — en cours Coeur de domaine Port / adapter de production
§2  Architecture hexagonale

§3  Le cycle de vie d'un fichier

De la reception au telechargement, gardes a chaque etape

Les octets transitent directement entre le client et le stockage (URL signee) : l'application ne relaie jamais les gros fichiers. L'analyse est asynchrone et declenchee automatiquement apres le depot.

1
Enregistrement
POST /api/files
Cree un FileRecord au statut PENDING et renvoie un ticket d'upload (URL signee, TTL court).
2
Depot direct
PUT → GCS
Le client televerse les octets directement vers le stockage. L'app ne les relaie pas.
3
Declenchement
GCS OBJECT_FINALIZE → Pub/Sub
La notification de fin d'upload est poussee vers /internal/scan-events. Aucune action manuelle.
4
Analyse
FileScanService → scanner externe
Appelle le service scanner (HTTP, OIDC) avec l'URI GCS ; celui-ci lit GCS et scanne via ClamAV. Le worker statue CLEAN ou INFECTED depuis la reponse et persiste le verdict.
5
Service
GET /api/files/{id}/content
Renvoie une redirection 302 vers une URL signee de telechargement, uniquement si CLEAN.
✓
Le garde-fou vit dans FileDownloadService : tout statut autre que CLEAN (en cours, echoue, et bien sur INFECTED) donne un refus. La regle est fail-safe : en cas de doute, on ne sert pas.
Schema 2 — Cycle de vie. En profils local/test, l'etape 3 est remplacee par un appel a /rescan (voir §4).

Un meme domaine, trois cablages

§5  Topologie de deploiement (GCP)

Trois services Cloud Run, une file au milieu

Le plan de service (api), le plan d'orchestration (worker) et le plan d'analyse (scanner) sont separes : ils s'echelonnent independamment, et une analyse lourde ne penalise jamais une requete de service. Le scanner est un service autonome (depot separe) ; il ne touche jamais la base.

Client / UI navigateur

Enregistre, depose et telecharge via des URLs signees — les octets ne transitent jamais par l'app.

api Cloud Run

Sert /api/** (public, X-API-Key verifie dans l'app). Signe les URLs, publie les demandes de scan.

app (profil gcp)

worker Cloud Run

Recoit le push Pub/Sub sur /internal/scan-events, appelle le scanner (HTTP, OIDC) et ecrit le verdict. Leger : min-instances=0 (scale-to-zero).

app (profil gcp)

scanner Cloud Run

Prive. POST /scan {gsUri} → lit GCS, scanne via ClamAV, renvoie le verdict. min-instances=1 (base de signatures chaude). Depot separe.

§6  Choix techniques & justifications

Ce qui a ete decide, et pourquoi

01

Architecture hexagonale

Le domaine ne depend d'aucun framework ; l'infrastructure implemente des ports. Pourquoi : tester le metier sans Spring ni base, et remplacer un adapter (stockage, scanner, file) sans toucher au domaine.

Testabilite · decouplage
02

URLs signees V4, octets hors de l'app

Le client depose et telecharge directement sur GCS via des URLs signees. Pourquoi : a l'echelle voulue (nombreux clients, fichiers tres volumineux), relayer les octets saturerait memoire et bande passante. L'app ne gere que des metadonnees.

Scalabilite · cout
03

Invariant de telechargement fail-safe

Le service ne renvoie le contenu que si le statut est CLEAN ; tout le reste est refuse. Pourquoi : c'est la garantie centrale du produit, et un defaut doit pencher du cote sur (ne pas servir) plutot que du cote risque.

Securite · garantie produit
04

Analyse asynchrone, scanner externalise

Le port ScanQueue decouple depot et analyse (en prod Pub/Sub + DLQ + reessais). Le scan est delegue a un service antivirus autonome (Python + ClamAV, depot separe) appele en HTTP (OIDC), qui seul lit GCS ; le worker seul ecrit le verdict. Pourquoi : encaisser les pics, isoler les analyses lentes, et cloisonner l'antivirus (aucun acces base, aucun faux « CLEAN » usurpable).

Resilience · cloisonnement
05

Profils sans dependance externe, domaine POJO

local et test tournent en memoire (base et JPA desactives par defaut) ; gcp active les adapters reels. Le domaine n'a aucune annotation JPA — des entites dediees et Flyway gerent la persistance. Pourquoi : dev et CI rapides et deterministes (sans Docker ni cloud), sans laisser le schema de base dicter le modele metier.

Vitesse · purete du modele
06

Cle API multi-tenant & reconciliation

Authentification machine-a-machine par X-API-Key (seul le hash SHA-256 est stocke, isolation par owner). ReconciliationService fait expirer les uploads restes trop longtemps en attente. Pourquoi : usage systeme-a-systeme sans secret en clair, et aucun fichier bloque indefiniment.

Securite · robustesse
